Why not analog capture?

  • 40–80 ms latency — unplayable for games
  • S-Video is blurry, chain of adapters
  • RetroTINK costs 300 EUR, still wired

Why bus snooping?

  • Read digital data before it becomes analog
  • Passive — zero impact on C64
  • Per-scanline VIC-II state capture
  • No video encoding/decoding needed

1. Bus Snooping (Pi Pico)

PIO reads C64 expansion port every PHI2 cycle (1 MHz). Captures writes to screen RAM, color RAM, bitmap, VIC-II registers. Builds shadow frame buffer.

2. SPI Transfer

Pico sends each scanline to Pi Zero via SPI @ 10 MHz. ~140 bytes/row: screen + color + bitmap + VIC-II deltas.

3. Wi-Fi Transmission

Pi Zero sends each row as raw UDP. No codec. 140 bytes = single Ethernet frame. Total ~1.4 MB/s @ 50 fps.

4. Framebuffer Render

Receiver writes rows to /dev/fb0. HDMI reads at 60 Hz. Latency: 0–16.7 ms per row (avg ~8 ms) depending on HDMI sync.

Latency: Pico reads row 0.064ms → SPI 0.08ms → UDP 0.5ms → fb write 0.01ms → HDMI sync 0–16.7ms = ~1–17ms (avg ~9ms)

Demoscene Raster Effects

Per-scanline VIC-II register tracking preserves all raster tricks:

  • Raster bars (color splits per line)
  • FLD / FLI (flexible line distance)
  • Split-screen text/bitmap modes
  • Sprite multiplexing
  • DYCP scrollers
  • Open border tricks
  • VSP (variable screen position)

How raster capture works

Every VIC-II register write is tagged with the current raster line ($D012). The receiver replays register changes at the exact scanline, reproducing mid-frame color changes, mode switches, and scroll offsets.
